Most cloud migrations burn effort at the wrong end of the pipeline. Teams
deploy a lift-and-shift machine before they've answered the question the
migration exists to answer: *what should the target look like?* Not the
servers — the shape. Services, boundaries, data ownership, failure
domains.

<DiagramTour spec={legacyMigrationDiagram}>
  <TourFrame scene="overview">
    **Read the legacy. Reason about the shape. Emit a plan.** Three stages,
    one decision: what should the cloud version look like?
  </TourFrame>
  <TourFrame scene="ingest">
    **Ingest — structure first.** Classify every legacy module. Output is
    a graph of domain, dependency, and I/O surface — not a summary.
  </TourFrame>
  <TourFrame scene="reason">
    **Reason — patterns, not vibes.** LLM maps the graph onto known
    patterns (strangler, anti-corruption layer, read-side projection).
    Every decision cites the legacy code it came from.
  </TourFrame>
  <TourFrame scene="plan">
    **Plan — runbooks, not slide decks.** IaC scaffolding, service
    boundaries, migration waves, data-gravity calls. Executable output.
  </TourFrame>
</DiagramTour>

The interesting part is not the move. The interesting part is the
*decision about what to move into*. This pipeline makes that decision
defensible.
